#　ソフトウェア工学 2026

##  ソフトウェア工学の基礎
ソフトウェア工学の目的は、ソフトウェア開発における「品質（Quality）」、「コスト（Cost）」、「納期（Delivery）」の最適なバランス（QCD）を実現するための手法や方法論を確立することである。ソフトウェアはハードウェアと異なり物理的な劣化はしない一方で、ビジネス要件の変化や環境の進化に適応し続けなければ陳腐化してしまう。IT人材不足や社会インフラとしての重要性が増す中、個人のプログラミングスキルだけでなく、チーム全体で共有できるエンジニアリングの知見が不可欠となっている。

## ソフトウェアライフサイクルと要求分析
ソフトウェアには誕生から運用、廃止に至る「ソフトウェアライフサイクル」がある。開発の初期段階である「要件定義」では、システムの目的や機能、システム構成、目標性能などを明確にし、プロジェクト関係者間でスコープを共有する。要件定義を終えた後はに設計、制作、テストとデバッグへと進むこととなる。
Hyrumの法則によればバグのないソフトウェアは存在しないと考えるべきであり、バグ累積曲線などを考慮し、いかに早期に欠陥を発見してコストを抑えるかが重要となる。

##  プロジェクト管理とWBS
プロジェクトとは、独自の目的を達成するための「有期性」を持つ業務であると言える。プロジェクト計画においては、インプット視点のフォアキャスティングではなく、目標から逆算して必要な道筋を立てる「バックキャスティング」のアプローチが求められる。
計画の具体化にはWork Breakdown Structureを用い、プロジェクトを細かい作業に分解する。このタスクの洗い出しは「モレなくダブりなく（MECE）」行うことが重要であり、これにより全体のスケジュール管理と作業計画を明確になる。

##  開発プロセスモデル
開発プロセスには、プロジェクトの規模や要求に応じて複数のアプローチが存在する。ウォーターフォール型は要件定義からテストまで順番に進めるモデルである。進捗管理が容易だが、後工程へのしわ寄せリスクがある。スパイラル型/反復型は、プログラム開発を細かく分割し、プロトタイプによるフィードバックを繰り返すものである。アジャイルプロセスでは、変化への対応を重視し、短いサイクルで動くソフトウェアを継続的に納品する。ほかにも、スクラム、カンバン、XPなどの手法がある。

##  コーディング規約と品質保持
ソフトウェア開発において「コードは書くよりも読まれることの方が多い」ため、可読性の高いコードを書くことが必須となる。例えばPythonではPEP8というスタイルガイドがあり、1行の長さ（79文字以内）、インデント（スペース4つ）、命名規則などが定められています。これらのルールをプロジェクト内で一貫して保つために、Flake8などのLinter（コードチェックツール）を導入することが推奨される。

##  バージョン管理とGit/GitHub
複数人で安全かつ効率的にコード開発を行うため、Gitなどの「分散型バージョン管理システム」が利用される。
ブランチ管理とは、作業を枝分かれさせて並行開発を行うための仕組みであり、メインブランチから開発用ブランチ（`dev`など）を切り出し、変更後にプルリクエストを通じてマージする「GitHubフロー」が広く用いられる。GitHubとは、リモートリポジトリのホスティングサービスであり、Issueによるタスク管理や、Projectsでのカンバンボードなど、開発を円滑にする機能がある。

##  CI/CDと自動化（GitHub Actions）
現代のソフトウェア開発では、開発のスムーズな進行と品質保証のためにCI/CDパイプラインが組み込まれる。CIとは、コード変更をトリガーとして自動テストやビルドを定期的に実行し、バグを早期発見するための仕組みである。CDは、テストを通過したコードを本番環境などに自動的にデプロイするための機能である。これらは、GitHub Actionsを使うことで、YAMLファイルに定義したジョブとステップをプッシュやプルリクエスト時に自動実行させることが可能となる。

##  GitHub Pagesによる情報公開
GitHub上のリポジトリの内容は、GitHub Pagesを利用することでWebページとして公開できる。
静的サイトジェネレータである「Jekyll」を組み合わせることで、開発者はMarkdownで文書を作成・コミットするだけで、CI/CDを利用することで自動的にHTMLへ変換・デプロイされる。
